# Scientific Writing Workflow

This file combines academic research discipline, Nature-style claim/evidence/boundary writing, and PaperSpine-style paper-spine planning.

## Non-Negotiable Principles

- User evidence comes first.
- Do not invent results, methods, sample sizes, statistics, figures, references, or novelty.
- Write the argument before writing sentences.
- Every important claim needs a user evidence anchor, a citation anchor, or a clear limitation.
- External papers teach evidence, context, and rhetorical structure; they do not replace the user's results.

## Intake Before Writing

Identify:

- target output: abstract, introduction, methods, results, discussion, cover letter, response letter, grant, thesis chapter, full manuscript;
- paper type: empirical research, methods paper, review, theoretical paper, registered report, grant/application;
- target venue or scene: journal, conference, thesis, funding application, internal report;
- core claim;
- user evidence: figures, tables, statistics, datasets, methods, code, observations;
- boundary: what the evidence does not show;
- required language and style;
- available citations and missing literature.

If core claim, evidence, or boundary is missing, expose the gap before drafting. A scaffold with placeholders is acceptable; invented content is not.

## Controlling Motivation

Before drafting, write one controlling motivation:

> This paper is worth reading because [specific gap/problem] remains unresolved, and the user's evidence shows [specific advance] within [boundary].

The motivation should be narrow enough that one paper can actually support it.

Avoid inflated motivations:

- "This study revolutionizes..."
- "This is the first ever..."
- "This fully explains..."

Prefer bounded motivation:

- "This study tests whether..."
- "This study characterizes..."
- "This study provides evidence that..."
- "This study distinguishes..."

## One-Sentence Argument

Use:

> In [system/problem], we show [advance] using [approach], supported by [evidence], with [boundary].

Examples:

- In first-grade children at learning risk, we characterize how inhibitory-control demands are reflected in ERP microstate dynamics, supported by Go/No-Go condition contrasts and group-level temporal-map measures, while limiting interpretation to task-related neural dynamics rather than clinical diagnosis.
- In a memory task, we show that retrieval success is associated with later parietal positivity and improved model-based evidence accumulation, while avoiding causal claims about hippocampal mechanisms without imaging evidence.

## Source Inventory

Create or mentally maintain a source inventory:

| Source | Type | What it can support | What it cannot support |
|---|---|---|---|
| Figure/table/result | User evidence | Specific empirical claim | External generalization |
| Local paper | Citation | Literature context | User's result unless directly comparable |
| External paper | Citation | Mechanism/background/comparison | User-specific data claim |
| Code/output | Method/result provenance | Reproducibility and analysis logic | Theory without interpretation |

## Claim Register

For each claim, record:

| Claim | Evidence anchor | Citation anchor | Support level | Boundary |
|---|---|---|---|---|

Support level:

- direct;
- partial;
- background;
- speculative;
- unsupported.

Unsupported claims must be deleted, weakened, or marked as future work.

## Section Blueprint

Do not force every task into a fixed IMRaD template. Choose the structure that fits the target scene.

For empirical cognitive neuroscience papers, a common blueprint is:

- Title: signal domain, method, population, and contribution.
- Abstract: problem -> approach -> main result -> interpretation -> boundary.
- Introduction: domain importance -> gap -> method value -> study aim/hypotheses.
- Methods: participants -> task -> acquisition -> preprocessing -> measures -> statistics.
- Results: behavior -> neural primary result -> secondary/robustness results.
- Discussion: main finding -> mechanism -> comparison with literature -> limitations -> implications.

For reviews:

- scope -> search/selection logic -> taxonomy -> evidence synthesis -> contradictions -> gaps -> future directions.

For grants/applications:

- problem -> candidate's preparation -> project feasibility -> novelty -> expected contribution -> risk management.

## Writing Rationale Matrix

Before major drafting or restructuring, build a matrix. It can be brief in chat or detailed in a file.

| Row ID | Manuscript Unit | Function | Motivation Link | Literature/SOTA Pattern | Venue Norm | User Evidence/Citation Anchor | Planned Text Move | Final Check |
|---|---|---|---|---|---|---|---|---|

Rules:

- First row justifies the whole paper's controlling structure.
- Each later row maps to a real writing unit, not just a section name.
- Split units when the writing function changes.
- Avoid generic reasons such as "improve clarity".
- Explain why the move is scientifically and rhetorically needed.

## Paragraph Rules

Each paragraph should have one job:

- context;
- gap;
- approach;
- result;
- comparison;
- mechanism;
- implication;
- limitation.

If a paragraph has two jobs, split it.

First sentence should state the paragraph's job. Later sentences should relate by cause, contrast, evidence, example, boundary, or implication.

## Verb Calibration

- show/demonstrate: strong direct evidence;
- reveal: direct evidence, often descriptive;
- indicate/suggest: indirect or moderate evidence;
- may/could: plausible mechanism or weak evidence;
- is consistent with: literature-aligned but not decisive;
- provides preliminary evidence: exploratory or limited evidence.

## Integrity Audit

Before finalizing, check:

- unsupported claims;
- fabricated or unverified citations;
- citations that do not support the sentence;
- results stated more strongly than statistics allow;
- causal language for correlational evidence;
- missing limitations;
- missing alternative explanations;
- paragraph jobs mixed together;
- terminology inconsistency;
- claims not connected to figures/tables/results.
